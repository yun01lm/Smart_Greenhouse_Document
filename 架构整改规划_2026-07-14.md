# 智慧大棚AIoT系统 — 架构整改规划

> **角色**：高级架构工程师 (SAE)  
> **日期**：2026-07-14  
> **依据**：首席架构师审查报告 v1.0  
> **原则**：
> 1. 保持项目现有形态稳定，不大拆大改
> 2. 优先修正影响后续开发的技术债务
> 3. 硬件不可用 → 允许设备模拟器，但协议格式必须和真实设备一致

## 第一部分：当前架构健康度评价

### 总体评价：**B+（可继续开发，需定向整改）**

架构骨架正确，核心设计经得起推敲。但存在 **3 个 P0 阻塞项** 和若干技术债务。当前最紧迫的问题不是代码质量，而是：

- **硬件端完全缺失** — 整个 IoT 系统的"物"端不存在，无法闭环验证
- **数据层不完整** — 3 张设计文档定义的表无 JPA 实体，阻塞后续 G04/G08 功能
- **配置化不足** — AI 参数和融合权重硬编码，论文实验无法做对比论证

### 各维度健康度

| 维度 | 评分 | 健康度 | 关键风险 |
|------|------|--------|----------|
| 分层架构 | 14/20 | 🟡 良好 | AlertController 直接操作 Repository |
| 数据库一致性 | 10/20 | 🟠 需关注 | 5 张表缺 JPA 实体，无 Flyway |
| AI 架构 | 13/20 | 🟡 良好 | Prompt/参数硬编码，无降级 |
| IoT 架构 | 8/20 | 🔴 严重 | ESP32 固件缺失，MQTT Topic 分散 |
| 可维护性 | 7/20 | 🟠 需关注 | 零测试，日志不规范 |

> **说明**：综合评分 52/100 看似很低，但这是因为 IoT 架构和可维护性两项拉分严重。核心的后端分层和 AI 策略设计是合格的。大学创新项目达到此水平已属中上。

### P0-1：设备模拟器

**1. 修改目标**
开发一个 Python/Shell 脚本，模拟 ESP32 设备通过 MQTT 协议向系统上报传感器数据，使系统可以在无真实硬件的条件下闭环运行和演示。

**2. 修改原因**
- ESP32 硬件不可用是客观约束，但系统必须有数据源
- 所有功能（实时监控、告警、多模态融合、AI 诊断）都依赖传感器数据
- 大创验收/比赛演示必须有可运行的系统

**3. 修改范围**
- 新建：`simulator/device_simulator.py` — MQTT 设备模拟器
  - 读取 `simulator/devices.json` 配置（设备列表、传感器类型、上报间隔）
  - 使用真实 MQTT Topic 格式：`greenhouse/{greenhouseId}/device/{deviceSn}`
  - 使用真实数据格式：与 `SensorData` 实体字段一致
  - 支持模拟传感器（温度、湿度、光照、CO2、土壤湿度、土壤温度）
  - 数据在一定范围内随机波动，模拟真实环境
- 新建：`simulator/devices.json` — 模拟设备配置文件
- 不改动任何后端代码

**4. 是否影响已有功能**
**不影响**。模拟器是独立的外部进程，与后端通过标准 MQTT 协议通信，后端完全无感知数据来源是真实硬件还是模拟器。

**5. 验证方式**
- 启动模拟器，观察 MQTT Broker 是否收到消息
- 检查 InfluxDB 中是否有传感器数据写入
- 打开 Web 端实时监控页面，确认数据图表正常刷新
- 确认告警引擎能基于模拟数据触发告警

### P0-2：创建 KnowledgeDocument JPA 实体

**1. 修改目标**
创建 `KnowledgeDocument` JPA 实体和 `KnowledgeDocumentRepository`，使知识库文档的元数据存储在 MySQL 中，向量数据仍存储在 Chroma 中。实现"MySQL 管元数据，Chroma 管向量"的架构。

**2. 修改原因**
- 当前文档元数据仅存 Chroma metadata，无法在 MySQL 中查询"有哪些文档已向量化"
- 阻塞 TASK-G04 知识库管理（文档上传、列表、删除、重新向量化）
- 论文中需要展示知识库文档管理体系

**3. 修改范围**
- 新建：`backend/.../entity/KnowledgeDocument.java`
  - 字段：id, title, category, filePath, fileType, fileSize, chunkCount, vectorIndexed, indexedAt, createdAt, updatedAt
  - 与设计文档 `knowledge_documents` 表结构一致
- 新建：`backend/.../repository/KnowledgeDocumentRepository.java`
  - 方法：findByCategory, findByVectorIndexed, countByCategory
- 修改：`backend/.../service/ChromaRetrievalService.java`（或新建 KnowledgeDocumentService）
  - 向量化完成后同步写入 MySQL
- 不改动 Chroma 存储逻辑

**4. 是否影响已有功能**
**不影响**。Chroma 中的已有向量数据保持不变。新增 MySQL 写入是追加操作，不影响现有 RAG 问答流程。

**5. 验证方式**
- 上传一个新文档，检查 MySQL `knowledge_documents` 表是否有记录
- 通过 API 查询文档列表，确认返回元数据
- RAG 问答功能不受影响

### P0-3：AlertController 分层修复

**1. 修改目标**
将 `AlertController` 中直接操作 `AlertRuleRepository` 和 `AlertLogRepository` 的代码提取到 `AlertService`，恢复 Controller→Service→Repository 分层。

**2. 修改原因**
- 明确违反项目分层约定，其他 Controller（DeviceController、UserController 等）均遵循此约定
- 修复成本极低（提取方法），无任何风险

**3. 修改范围**
- 修改：`backend/.../controller/AlertController.java`
  - 移除对 Repository 的直接注入
  - 所有业务逻辑改为调用 AlertService
- 修改：`backend/.../service/AlertService.java`（或新建 AlertManagementService）
  - 新增方法：listRules, createRule, updateRule, deleteRule, listLogs
- 不改动 API 端点路径和响应格式

**4. 是否影响已有功能**
**不影响**。API 接口和响应格式完全不变，仅内部实现从 Controller 移到 Service。

**5. 验证方式**
- 调用告警规则 CRUD API，确认功能正常
- 调用告警日志查询 API，确认功能正常
- 代码审查确认 Controller 不再注入 Repository

### P0-4：多模态融合配置外部化

**1. 修改目标**
将 `VisualHealthCalculator` 和 `EnvironmentHealthCalculator` 中硬编码的权重、阈值、修正因子提取到 `application.yml`，使用 `@ConfigurationProperties` 绑定。

**2. 修改原因**
- 直接影响论文可论证性 — 当前无法做"不同权重下的评分对比实验"
- 配置化后可以通过修改 YAML 快速做 A/B 测试，无需重新编译
- 不改变任何业务逻辑，仅改变值的来源

**3. 修改范围**
- 新建：`backend/.../config/FusionConfig.java` — `@ConfigurationProperties(prefix = "greenhouse.fusion")`
  - 环境权重（温度、湿度、光照、CO2、土壤湿度、土壤温度）
  - 环境/视觉融合比例（默认 60%/40%）
  - 各指标阈值（正常/警告/危险）
  - 天气风险修正因子
- 修改：`application.yml` — 新增 `greenhouse.fusion` 配置段
- 修改：`VisualHealthCalculator.java` — 权重从 FusionConfig 读取
- 修改：`EnvironmentHealthCalculator.java` — 权重/阈值从 FusionConfig 读取

**4. 是否影响已有功能**
**不影响**。配置默认值等于当前硬编码值，行为完全不变。

**5. 验证方式**
- 使用默认配置运行，确认评分结果与修改前一致
- 修改 YAML 中的权重（如环境:视觉 = 70:30），确认评分结果按预期变化
- 论文实验可直接用此机制做对比

- MQTT 订阅和发布功能正常

## 第五部分：整改影响评估

### 5.1 是否导致项目偏离最初方案？

**不会偏离。** 逐一对照：

| 最初方案要素 | 整改影响 | 说明 |
|-------------|----------|------|
| 三库架构（MySQL+InfluxDB+Chroma） | 无影响 | P0-2 新增 MySQL 表，强化而非削弱三库架构 |
| AI 策略模式抽象层 | 无影响 | P0-4/P1-3 仅将硬编码改为配置化，不改变接口设计 |
| 多模态融合评分 | 增强 | P0-4 配置化后反而更灵活，可做对比实验 |
| 统一设备模型（devices 表） | 无影响 | 已拒绝 s/a 分离建议，保持现有设计 |
| 分层架构（Controller→Service→Repository） | 修复 | P0-3 修复唯一的分层违规点 |
| MQTT 协议通信 | 无影响 | P0-1 模拟器遵循真实协议，P1-5 仅统一常量 |
| Web 端 Vue3+Element Plus | 无影响 | 所有整改仅涉及后端 |
| Android APP | 无影响 | 不涉及 |

### 5.2 是否导致论文创新点降低？

**不会降低，反而增强。** 关键论文创新点分析：

| 论文创新点 | 整改前状态 | 整改后状态 |
|-----------|-----------|-----------|
| 多模态融合评分算法 | 权重硬编码，无法做对比实验 | **配置化后可以做权重对比实验** ✅ |
| AI 策略模式设计 | 已实现，但 Prompt 不可调 | **Prompt 配置化后可论证 Prompt 工程** ✅ |
| RAG 知识检索增强 | 已实现，但无文档管理 | **KnowledgeDocument 实体补全文档管理** ✅ |
| 诊断准确率论证 | 无历史数据积累 | **diagnostic_records 写入后可展示准确率曲线** ✅ |
| 长势评估模型 | 评估后无持久化 | **GrowthAssessment 实体可展示历史趋势** ✅ |
| 统一设备抽象模型 | 已实现 | 保持不变（拒绝 s/a 分离建议，保留此创新点）✅ |

### 5.3 是否导致商业化扩展能力下降？

**不会下降。** 分析：

| 商业化能力 | 整改影响 | 说明 |
|-----------|----------|------|
| 设备模拟器 | **增强** | 模拟器本身就是商业化 Demo 的标配，客户演示不需要搬硬件 |
| 配置化 | **增强** | 权重/Prompt 可配置是 SaaS 多租户的基础 |
| 数据积累 | **增强** | 诊断记录和评估记录持久化，是数据驱动优化的基础 |
| 分层修复 | **增强** | 架构一致性降低新员工上手成本 |
| 拒绝过度工程 | **增强** | 不引入 Flyway/设备影子/Swagger 等当前无用的企业级组件，保持项目轻盈 |

### 5.4 明确拒绝的修改及理由

| 拒绝项 | 理由 |
|--------|------|
| sensors/actuators 分离 | 统一设备模型是**有意识的设计决策**，通过 `device_type` 区分。论文可论证"统一设备抽象模型"作为创新点。分离反而增加代码复杂度 |
| Flyway 数据库迁移 | JPA `ddl-auto: update` 对大学项目完全够用。Flyway 需要 25 个迁移脚本的维护成本，ROI 极低 |
| LLM 降级策略 | 只有 1 个 LLM API Key，无备用 Provider。降级逻辑无实际可用目标，属于无效代码 |
| Swagger 自动文档 | 18 个 API 文档已手写完成且质量合格，引入 SpringDoc 需额外注解维护 |
| 前端服务端分页 | 数据量在百级以内，前端分页完全够用。引入服务端分页增加前后端复杂度 |
| 单元测试 | 核心目标是功能完整+论文论证，不是代码覆盖率。时间有限 |
| Pinia 状态管理 | 当前组件数量少，props 传递够用。后续组件增多时再引入 |
| 设备影子 | 企业级 IoT 平台特性，大学项目不需要 |
