# 测试用例模板

> 本文档定义智慧大棚AIoT系统的测试用例编写规范和分类说明。

---

## 测试用例模板格式

每个测试用例使用以下表格格式编写：

| 字段 | 说明 |
|------|------|
| **用例ID** | 唯一标识，格式：`{模块缩写}-{编号}`（如 `AUTH-001`、`GH-003`） |
| **测试场景** | 简要描述测试的业务场景 |
| **前置条件** | 执行测试前必须满足的条件（如已登录、已有大棚数据等） |
| **测试步骤** | 详细的操作步骤，每步一行 |
| **预期结果** | 每一步对应的预期输出或系统行为 |
| **测试数据** | 测试中使用的具体数据（可选） |
| **优先级** | P0（核心功能）/ P1（重要功能）/ P2（一般功能） |
| **测试类型** | 单元测试 / 集成测试 / 端到端测试 / 传感器精度测试 / 通信稳定性测试 |
| **关联模块** | 该测试关联的后端模块编号（如 C1、C3） |

## 完整示例测试用例

### GH-001 - 大棚创建功能测试

| 字段 | 内容 |
|------|------|
| **用例ID** | GH-001 |
| **测试场景** | 棚主用户创建新大棚 |
| **前置条件** | 用户已注册为棚主(OWNER)角色，已登录获得有效JWT Token |
| **优先级** | P0 |
| **测试类型** | 集成测试 |
| **关联模块** | C1 用户认证、C3 大棚管理、C18 多角色权限 |

**测试步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 发送POST请求到 `/api/v1/greenhouses`，请求体包含大棚名称、位置、作物类型 | HTTP 200，返回创建的Greenhouse对象，包含id、name、location、crop_type等字段 |
| 2 | 验证返回数据中的 `user_id` 是否为当前登录用户 | `user_id` 等于当前用户的ID |
| 3 | 发送GET请求到 `/api/v1/greenhouses` 查看大棚列表 | 列表中包含刚创建的大棚记录 |
| 4 | 使用员工(WORKER)角色Token发送相同创建请求 | HTTP 403，返回错误码3004（仅棚主可执行此操作） |
| 5 | 发送POST请求，name字段为空 | HTTP 400，返回参数校验错误信息 |
| 6 | 发送POST请求，name字段超长（>100字符） | HTTP 400，返回参数校验错误信息 |

**测试数据**：

```json
{
  "name": "3号番茄大棚",
  "location": "河北省石家庄市",
  "crop_type": "番茄",
  "province": "河北省",
  "city": "石家庄市",
  "district": "藁城区",
  "town": "岗上镇",
  "village": "大同村"
}
```

### ALERT-001 - 预警触发与推送测试

| 字段 | 内容 |
|------|------|
| **用例ID** | ALERT-001 |
| **测试场景** | 传感器数据超过预警阈值时系统自动触发预警并推送通知 |
| **前置条件** | 1. 大棚已创建并绑定ESP32设备；2. 已配置温度预警规则（上限35°C）；3. WebSocket连接已建立 |
| **优先级** | P0 |
| **测试类型** | 端到端测试 |
| **关联模块** | C4 MQTT消费、C5 时序数据、C6 环境预警、C11 WebSocket推送 |

**测试步骤**：

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 通过MQTT模拟器向 `greenhouse/1/group/1/device/dev001/sensor/temp` 发送温度值36.5°C | MQTT消息被后端接收并存入InfluxDB |
| 2 | 检查预警记录表 `alerts` | 新增一条WARNING级别预警记录，title为"温度过高预警" |
| 3 | 检查WebSocket推送 `/topic/greenhouse/1/alerts` | 客户端收到预警推送消息，包含传感器组定位(group_id=1) |
| 4 | 发送温度值34.0°C（低于阈值） | 不触发新预警，系统恢复正常 |
| 5 | 标记预警为已读 PUT `/api/v1/alerts/{id}/read` | HTTP 200，预警read_status变为已读 |

## 测试分类说明

### 1. 单元测试 (Unit Test)

**定义**：针对单个类或方法的独立测试，不依赖外部系统。

**测试范围**：

- Service层业务逻辑
- 工具类方法
- 实体类转换逻辑
- AI抽象层接口契约

**工具**：JUnit 5 + Mockito

**覆盖率要求**：

| 层级 | 最低覆盖率 |
|------|-----------|
| Service层 | 80%+ |
| AI抽象层 | 100%（接口契约） |

**命名规范**：`{方法名}_{场景}_{预期结果}`

**示例**：

```java
@Test
void createGreenhouse_ValidInput_ShouldSucceed() { }
void createGreenhouse_WorkerRole_ShouldThrowException() { }
void createGreenhouse_NullName_ShouldThrowException() { }
```

---

### 2. 集成测试 (Integration Test)

**定义**：测试多个组件之间的交互，包括Controller、Service、Repository的协作。

**测试范围**：

- Controller层API接口
- Repository层数据访问
- 数据库交互（使用H2内存库）

**工具**：Spring Boot Test + MockMvc + @DataJpaTest

**覆盖率要求**：

| 层级 | 最低覆盖率 |
|------|-----------|
| Controller层 | 60%+ |
| Repository层 | 50%+（自定义查询方法必须测试） |

**必须覆盖的场景**：

1. 正常场景：合法输入 → 预期输出
2. 边界场景：空值/null/最大值/最小值
3. 异常场景：权限不足/资源不存在/参数非法
4. 外部依赖异常：AI API超时/MQTT断连/数据库异常

---

### 3. 端到端测试 (End-to-End Test)

**定义**：模拟真实用户操作流程的完整测试，覆盖从前端到后端的全部链路。

**测试范围**：

- APP端操作流程（如：登录→查看大棚→查看实时数据→控制设备）
- Web端操作流程（如：管理员登录→创建用户→配置预警规则→查看报表）
- 专家咨询全流程（用户求助→专家接单→聊天→结束）

**工具**：Playwright（Web端）/ Espresso或手动测试（APP端）

**典型场景**：

1. 用户注册→登录→创建大棚→绑定设备→查看实时数据
2. 传感器数据→MQTT上报→InfluxDB存储→WebSocket推送→APP实时更新
3. 环境异常→预警触发→推送通知→用户查看→手动控制设备
4. 用户拍照→图片诊断→低置信度→一键求助专家→专家接单→实时聊天

---

### 4. 传感器精度测试

**定义**：验证各传感器读数精度的专项测试。

**测试范围**：

| 传感器 | 型号 | 精度要求 | 测试方法 |
|--------|------|----------|----------|
| 空气温湿度 | SHT30 | ±0.3°C / ±2%RH | 与标准温湿度计对比 |
| 光照强度 | BH1750FVI | ±20% | 与标准光照计对比 |
| CO2浓度 | MQ-135 | ±10% | 与标准CO2检测仪对比 |
| O2浓度 | O2传感器模块 | ±5% | 与标准O2检测仪对比 |
| 土壤湿度 | 电容式 | ±5% | 烘干称重法对比 |
| 土壤温度 | DS18B20 | ±0.5°C | 与标准温度计对比 |
| 土壤EC | 电导率探头 | ±10% | 与标准EC计对比 |
| NPK | RS485五合一 | ±10% | 与实验室检测对比 |

**测试频率**：初次部署时全面测试，之后每季度校准一次。

---

### 5. 通信稳定性测试

**定义**：验证系统各通信链路的稳定性。

**测试范围**：

| 通信链路 | 测试项 | 通过标准 |
|----------|--------|----------|
| ESP32 ↔ WiFi | 连接稳定性 | 24小时不掉线，断开后30秒内自动重连 |
| ESP32 ↔ MQTT Broker | 消息可靠性 | 1000条消息无丢失，QoS=1 |
| MQTT Broker ↔ Spring Boot | 消息消费延迟 | P99 < 500ms |
| Spring Boot ↔ InfluxDB | 写入性能 | 每秒100条传感器数据写入无丢失 |
| WebSocket连接 | 断线重连 | 断开后5秒内自动重连，消息不丢失 |
| API接口 | 响应时间 | P99 < 1000ms（普通查询），P99 < 3000ms（AI调用） |
| frp内网穿透 | 连接稳定性 | 72小时无断连 |

**测试方法**：

- 使用MQTT压力测试工具模拟传感器数据上报
- 使用JMeter进行API压力测试
- 监控WebSocket连接数和断开重连次数
- 记录并分析系统日志中的异常

## 测试禁止事项

- **禁止**将测试代码提交到生产分支时测试不通过
- **禁止**在测试中使用真实的外部API（必须Mock）
- **禁止**测试之间相互依赖（每个测试独立运行）
- **禁止**在测试中硬编码环境相关的值（端口、路径等）
- **禁止**测试用例无断言或仅有 `assertTrue(true)` 占位
